Email phishingowy przeszedł weryfikację DKIM i DMARC, ponieważ atakujący przejął konto SendGrid firmy arterexmedical[.]com - prawdziwej firmy medycznej. Standardowe filtry antyphishingowe oparte na reputacji domeny nie wykryją tego ataku.
Executive Summary

W dniu 21 kwietnia 2026 roku odnotowano próbę phishingową wymierzoną w użytkownika firmowego Microsoft 365. Atak wykorzystuje OAuth Device Code Flow (RFC 8628) - legalny mechanizm autoryzacji zaprojektowany dla urządzeń IoT - do wyłudzenia tokenów dostępu do konta Microsoft bez konieczności przechwytywania hasła lub kodu MFA.

Łańcuch ataku składa się z trzech warstw: fałszywy email powiadomienia Teams dostarczony przez skompromitowane konto SendGrid (arterexmedical[.]com), przekierowanie przez skompromitowaną stronę singapurską (leoch[.]sg), oraz fałszywy interfejs Teams hostowany na Cloudflare Workers. Kulminacją jest wyświetlenie ofierze kodu urządzenia i nakłonienie jej do autoryzacji go na prawdziwej stronie Microsoft - co oddaje atakującemu pełny dostęp do konta przez access token i 90-dniowy refresh token.

Łańcuch ataku - 3 etapy
01
Email Delivery - kompromitacja SendGrid arterexmedical[.]com
Email nadawany jest z info[@]arterexmedical[.]com - legitymnej firmy medycznej, której konto SendGrid zostało przejęte przez atakującego. Dzięki temu email przechodzi weryfikację DKIM PASS i DMARC PASS, trafiając do skrzynki ofiary bez ostrzeżeń.

Rzeczywisty nadawca: 168.245.102.229 (SendGrid - xvfrsszr.outbound-mail.sendgrid[.]net), routowany przez Mimecast UK (91.220.42.227). SPF pada na poziomie Microsoft (FAIL), ale DMARC jest spełniony przez alignment DKIM - email zostaje dostarczony.

Treść emaila: imitacja powiadomienia Microsoft Teams - "Hi [imię], please take a look at the shared document and provide your feedback when you get a chance." z przyciskiem Open Message.
02
Redirect Chain - leoch[.]sg → Cloudflare Workers
Kliknięcie przycisku uruchamia łańcuch przekierowań:

SendGrid tracking URL → hxxps://leoch[.]sg/about-us/#[email-ofiary] → hxxps://secure-link-ek3t[.]c-kzevzz5a[.]workers[.]dev/l/7vmjNVmDDOs#[email-ofiary]

Domena leoch[.]sg należy do singapurskiego producenta baterii - prawdopodobnie skompromitowany WordPress lub skradzione dane dostępowe. Na stronie osadzony jest minimalistyczny JavaScript inicjujący fetch do właściwej infrastruktury phishingowej po 2-sekundowym opóźnieniu, symulującym ładowanie Teams.

Email ofiary przekazywany jest w hash URL (#email@domain.com) - fragment nie jest wysyłany do serwera, co utrudnia detekcję przez URL scanery.
03
OAuth Device Code Abuse - wyłudzenie tokenu Microsoft 365
Strona Cloudflare Workers wyświetla fałszywy interfejs Microsoft Teams z komunikatem o udostępnionym pliku Excel oraz kodem weryfikacyjnym (GWX74K3UC). Instrukcje nakłaniają ofiarę do kliknięcia "Open File", co otwiera prawdziwy portal Microsoft (login[.]microsoftonline[.]com/common/oauth2/deviceauth).

Po wpisaniu kodu ofiara widzi: "Logujesz się do aplikacji Microsoft Office na innym urządzeniu znajdującym się w lokalizacji Stany Zjednoczone." Akceptując - nieświadomie autoryzuje aplikację atakującego, który w tym momencie odbiera access token (1h) oraz refresh token (90 dni).

🔍
Kluczowy insight - dlaczego DMARC przeszedł mimo SPF fail?
DMARC wymaga alignment albo SPF albo DKIM z domeną From:. Atakujący, kompromitując konto SendGrid firmy arterexmedical[.]com, uzyskał dostęp do kluczy prywatnych DKIM tej domeny - co pozwoliło podpisać email ważną sygnaturą. SPF padło (inny IP), ale DKIM alignment wystarczył do spełnienia polityki DMARC. To właśnie dlatego kompromitacja legitymnej domeny jest cenniejsza niż spoofing.

Infrastruktura phishingowa
INF-1
leoch[.]sg - skompromitowana strona korporacyjna
Leoch International Technology Limited to rzeczywisty singapurski producent baterii. Strona najprawdopodobniej oparta na WordPressie z niezaktualizowanymi pluginami lub skradzionymi danymi dostępowymi. Atakujący wstrzyknął minimalny JavaScript do strony /about-us/, który po 2 sekundach wykonuje fetch do głównej infrastruktury phishingowej i nadpisuje zawartość strony odpowiedzią - technika pozwalająca uniknąć przekierowania widocznego w pasku adresu.
INF-2
Cloudflare Workers - hosting strony phishingowej
Właściwa strona phishingowa hostowana na secure-link-ek3t[.]c-kzevzz5a[.]workers[.]dev. Cloudflare Workers popularyzowane przez atakujących ze względu na: darmowy certyfikat TLS (brak red flagi w przeglądarce), trudność takiedownu (wymaga raportu nadużycia do Cloudflare), natychmiastowy redeploy po usunięciu oraz losowy subdomain (c-kzevzz5a) utrudniający blokadę.

Mechanizm OAuth Device Code Flow Abuse

Legalny Device Code Flow (RFC 8628) - punkt wyjścia:

Mechanizm zaprojektowany dla urządzeń bez przeglądarki (smart TV, CLI, IoT). Urządzenie generuje kod przez API Microsoft i wyświetla go użytkownikowi, który wpisuje go na microsoft.com/devicelogin z osobnego urządzenia. Po autoryzacji urządzenie odbiera token. Kluczowa luka: brak mechanizmu weryfikacji, co faktycznie autoryzuje użytkownik.

01
Generowanie kodu urządzenia przez atakującego
Atakujący rejestruje aplikację w Azure AD (lub przejmuje istniejącą) i odpytuje Microsoft o device code. W odpowiedzi otrzymuje device_code (dla serwera) oraz user_code (dla ofiary). Jednocześnie rozpoczyna cykliczne odpytywanie endpointu tokenowego - czekając aż ofiara autoryzuje kod.
02
Wyświetlenie kodu ofierze - fałszywy interfejs Teams
Strona Workers wyświetla user_code (GWX74K3UC) jako "Verification code" z instrukcjami krok po kroku. Całość oprawiona jest w realistyczny interfejs Teams z ikoną Excel i fikcyjną wiadomością o udostępnionym dokumencie - budując zaufanie i poczucie pilności.
03
Autoryzacja na prawdziwej stronie Microsoft
Kliknięcie "Open File" otwiera legalną stronę Microsoft (login[.]microsoftonline[.]com/common/oauth2/deviceauth). Ofiara loguje się swoimi danymi - włącznie z MFA - i wpisuje kod. Widoczny komunikat: "Logujesz się do aplikacji Microsoft Office na innym urządzeniu znajdującym się w lokalizacji Stany Zjednoczone."

Geolokacja USA wskazuje lokalizację serwera atakującego. Nazwa "Microsoft Office" to prawdopodobnie fałszywa lub przejęta rejestracja aplikacji w Azure AD.
04
Odbiór tokenów - pełny dostęp do konta
W momencie zatwierdzenia przez ofiarę serwer atakującego odbiera:

Access Token - ważny 1h, natychmiastowy dostęp do Microsoft Graph API (email, pliki, Teams, kontakty)
Refresh Token - ważny 90 dni domyślnie, pozwala na ciągłe odnawianie dostępu. Przeżywa zmianę hasła.

Zakres uprawnień zależy od rejestracji aplikacji: prawdopodobnie Mail.ReadWrite, Files.ReadWrite.All, Contacts.Read, User.Read.
💡
Dlaczego MFA nie pomaga?
MFA chroni przed kradzieżą hasła. W tym ataku ofiara samodzielnie przechodzi przez pełny proces uwierzytelniania - włącznie z MFA - na prawdziwej stronie Microsoft. Atakujący nie przechwytuje kodu MFA ani sesji cookie. Uzyskuje token OAuth, który jest już wynikiem zakończonego uwierzytelniania.

Detekcja

Microsoft Sentinel / Entra ID - KQL: Device Code Flow z nieoczekiwanej lokalizacji:

// [HIGH] OAuth Device Code Flow - uwierzytelnienie spoza oczekiwanej lokalizacji
// MITRE: T1528 · T1550.001
SigninLogs
| where TimeGenerated > ago(30d)
| where AuthenticationProtocol == "deviceCode"
| extend Country = tostring(LocationDetails.countryOrRegion)
| where Country !in ("Poland", "Germany")  // dostosuj do swojego środowiska
| where ResultType == 0
| project
    TimeGenerated,
    UserPrincipalName,
    AppDisplayName,
    Country,
    IPAddress,
    DeviceDetail,
    ResultType
| order by TimeGenerated desc

Microsoft Sentinel - KQL: Nowe OAuth consent granty z szerokim zakresem:

// [MEDIUM] Nowe zgody OAuth dla aplikacji z szerokim zakresem uprawnień
// MITRE: T1528 · T1098.001
AuditLogs
| where TimeGenerated > ago(7d)
| where OperationName == "Consent to application"
| extend AppName = tostring(TargetResources[0].displayName)
| extend Permissions = tostring(TargetResources[0].modifiedProperties)
| where Permissions has_any ("Mail.", "Files.", "User.ReadWrite", "Contacts.")
| project TimeGenerated, InitiatedBy, AppName, Permissions

Microsoft 365 Defender - KQL: Email imitujący Teams z linkiem do workers.dev:

// [MEDIUM] Phishing email imitujący Teams - link do Cloudflare Workers
// MITRE: T1566.002
EmailEvents
| where TimeGenerated > ago(7d)
| where Subject has_any ("Teams", "Shared", "Document", "Workflow", "Review")
| join kind=inner (
    EmailUrlInfo
    | where Url has_any ("workers.dev", "deviceauth", "leoch.sg")
) on NetworkMessageId
| project TimeGenerated, Subject, SenderFromAddress, RecipientEmailAddress, Url, ThreatTypes
ℹ️
Uwaga operacyjna
Reguły KQL dla SigninLogs wymagają licencji Azure AD P2 lub Microsoft Sentinel z odpowiednim konektorem danych. Reguła consent grant jest szczególnie wartościowa - nowe OAuth consent z szerokim zakresem uprawnień powinny zawsze wymagać manualnej weryfikacji przez zespół bezpieczeństwa.

Mitigacja
1. Conditional Access - ograniczenie Device Code Flow
Azure AD → Security → Conditional Access → New Policy
Client apps: Mobile apps and desktop clients | Locations: Exclude trusted | Grant: Block
Wdrożyć najpierw w trybie Report-only, zweryfikować false positive baseline przed aktywacją.
2. Ograniczenie zgód użytkownika (Application Consent Policies)
Azure AD → Enterprise Applications → Consent and permissions
User consent: Do not allow user consent | Admin consent requests: Enable
Zapobiega autoryzacji nieznanych aplikacji bez wiedzy zespołu bezpieczeństwa.
3. Działania po kompromitacji - jeśli token został wydany
Revoke sessions: Azure AD → Users → [User] → Revoke all refresh tokens
Audit OAuth grants: Enterprise Applications → przegląd ostatnich 90 dni
Sprawdź reguły skrzynki: Get-InboxRule -Mailbox [user] | Where {$_.ForwardTo}
Sprawdź forwarding: Get-Mailbox [user] | Select ForwardingAddress, ForwardingSMTPAddress
✅
Wskazówka dla użytkowników
Kody urządzenia OAuth powinny pojawiać się wyłącznie gdy samodzielnie inicjujesz logowanie na urządzeniu bez przeglądarki. Jeśli kod pojawia się w emailu lub na stronie internetowej - to phishing. Zawsze zwróć uwagę na komunikat o lokalizacji urządzenia w panelu autoryzacji - lokalizacja inna niż twoja powinna natychmiast wzbudzić podejrzenia.
