Aktywna kampania
🔴
Bypass MFA przez legalną infrastrukturę Microsoft
Atak omija wszystkie formy MFA - TOTP, push notification i klucze FIDO2. Ofiara autoryzuje kod urządzenia na prawdziwym portalu Microsoft - backend atakującego otrzymuje ważny access token (1h) i refresh token (90 dni) bez zobaczenia hasła ani odpowiedzi MFA. Token pozostaje ważny po zmianie hasła.
⚠️
Podwójna impersonacja marki + finansowy lure
Kampania łączy dwie impersonacje wysokiego zaufania: JPMorgan Chase (powiadomienie o zaszyfrowanym emailu Virtru) oraz Adobe Acrobat Sign (portal weryfikacji dokumentów). Nazwa dokumentu ACH_Payment-ISO 201202.pdf tworzy poczucie pilności - ACH (Automated Clearing House) to sieć przelewów bankowych w USA, celowo wymierzona w pracowników działów finansowych.
Streszczenie

W dniu 27 kwietnia 2026 roku użytkownik zgłosił podejrzany email podszywający się pod JPMorgan Chase & Co., dostarczający powiadomienie w stylu Virtru o zaszyfrowanej wiadomości. Link w emailu zawierał adres docelowy zakodowany w Base64, przekierowujący przez tracker mailingowy do fałszywego portalu Adobe Acrobat Sign. Aby "uzyskać dostęp do chronionego dokumentu", ofiara była nakłaniana do weryfikacji tożsamości z Microsoftem - klasyczne nadużycie OAuth 2.0 Device Authorization Grant (RFC 8628).

W odróżnieniu od ataków credential harvesting, na stronie phishingowej nie jest wprowadzane żadne hasło. Ofiara wchodzi w interakcję wyłącznie z legalną infrastrukturą Microsoft (login.microsoftonline[.]com/common/oauth2/deviceauth), co czyni atak praktycznie niewidocznym zarówno dla ofiary, jak i dla narzędzi bezpieczeństwa. Backend phishingowy odpytuje endpoint tokenów Microsoft co 3 sekundy do momentu autoryzacji, po czym po cichu przekierowuje ofiarę na prawdziwy dokument PDF JPMorgan jako kamuflaż - ofiara nie ma żadnego sygnału, że coś złośliwego się wydarzyło.

To drugi atak Device Code Phishing zaobserwowany w ciągu 6 dni - patrz THREAT-012 dla kampanii z lure'em Microsoft Teams. Wspólny mechanizm nadużycia OAuth przy różnej infrastrukturze lure sugeruje albo ten sam kit phishingowy z konfigurowalnymi szablonami, albo aktora aktywnie iterującego warianty impersonacji marki.

Łańcuch ataku - 5 etapów
01
Dostarczenie emaila - impersonacja JPMorgan Chase + Virtru
Email phishingowy imituje powiadomienie o zaszyfrowanej wiadomości Virtru wysłanej przez JPMorgan Chase. Virtru to legitymna usługa szyfrowania emaili stosowana przez instytucje finansowe - lure jest celowo dobrany dla odbiorców współpracujących z JPMC lub oczekujących bezpiecznej korespondencji finansowej.

Nadawca: noreplies.sua[@]jpmchase[.]com
Subdomena .sua jest anomalna - legitymne adresy noreply JPMorgan nie używają tego prefiksu subdomeny. Jest to adres sfałszowany lub zarejestrowane konto lookalike.

Treść emaila imituje wrapper Virtru: "is using Virtru to send and receive encrypted email" z przyciskiem CTA Unlock Message oraz ostrzeżeniem o wygaśnięciu dostępu po 90 dniach budującym poczucie pilności. Stopka poprawnie renderuje logo Virtru i copyright - klon wysokiej wierności.
02
Łańcuch przekierowań - tracker Base64 → domena phishingowa
Przycisk "Unlock Message" prowadzi do trackera URL na mailer.servidoraweb[.]net - domenie infrastruktury masowego mailingu. Parametr url zawiera Base64-zakodowany adres docelowy:

hxxps://mailer.servidoraweb[.]net/class/link.php?id=wweqsyoosepruswqy&url=aHR0cHM6Ly9icmVudGJybHR0b24uY29tLz9lbWFpbD1j[...]

Dekodowanie Base64 → hxxps://brentbrltton[.]com/?email=c[email-ofiary]

Warta uwagi rozbieżność domenowa: zakodowany URL zawiera brentbrltton[.]com (brakuje jednej litery 't'), podczas gdy przeglądarka wyświetla brentbritton[.]com - prawdopodobnie dwie osobno zarejestrowane domeny używane jako etapy przekierowania, co utrudnia pojedynczy takedown. Obie serwują ten sam kit phishingowy pod ścieżką /93c67ceedebf224c.

Adres email ofiary jest przekazywany jako parametr URL, wypełniając stronę phishingową z góry i umożliwiając atakującemu śledzenie per ofiara.
03
Strona lądowania - fałszywy portal Adobe Acrobat Sign
Strona pod adresem hxxps://brentbritton[.]com/93c67ceedebf224c renderuje przekonujący klon portalu Adobe Acrobat Sign:

• Logo i branding Adobe (Adobe Acrobat Sign w nagłówku przeglądarki i na stronie)
• Ikona zablokowanego dokumentu z nazwą pliku: ACH_Payment-ISO 201202.pdf
• "Verify your identity - To access this document, please complete verification with Microsoft."
• Dynamicznie generowany kod weryfikacyjny wyświetlony prominentnie (zaobserwowany: A2ND3XUK5)
• Przycisk Copy Code + przycisk Continue to Microsoft
• Stopka: "Secured by Microsoft" (zielona ikona kłódki) oraz "© 2026 Adobe. All rights reserved."

Nazwa dokumentu ACH Payment to celowy finansowy lure - ACH (Automated Clearing House) to sieć przelewów bankowych w USA, sugerująca nieudany lub oczekujący przelew, który pracownicy działu finansowego będą chcieli pilnie zweryfikować.
04
Nadużycie OAuth Device Code - kradzież tokenu przez microsoft.com
Kliknięcie Continue to Microsoft otwiera prawdziwy portal uwierzytelniania urządzenia Microsoft:
hxxps://login.microsoftonline[.]com/common/oauth2/deviceauth

Ofiara wkleja skopiowany kod i widzi: "Logujesz się do aplikacji Microsoft Authentication Broker na innym urządzeniu znajdującym się w lokalizacji Stany Zjednoczone."

Kończąc logowanie, ofiara nieświadomie autoryzuje aplikację atakującego, który w tym momencie otrzymuje:
• Access Token - ważny 1 godzinę, natychmiastowy dostęp do API Microsoft 365
• Refresh Token - ważny do 90 dni, cicho odnawia access tokeny

Backend atakującego zainicjował ten przepływ przez POST /api/device/start do endpointu autoryzacji urządzenia Microsoft, otrzymując parę device code / user code. Strona phishingowa wyświetla user code, podczas gdy backend co 3 sekundy (setInterval(ck, 3000)) odpytuje GET /api/device/status/{sid} w oczekiwaniu na autoryzację przez ofiarę.
05
Kamuflaż po kompromitacji - przekierowanie na prawdziwy PDF JPMorgan
Gdy kod urządzenia zostaje autoryzowany i /api/device/status zwraca status == 'completed', strona phishingowa po cichu przekierowuje ofiarę na:
hxxps://static.chasecdn[.]com/content/dam/legal-agreements/library/en/Wire_LA.pdf

To prawdziwy dokument JPMorgan Chase - PDF umowy prawnej Wire Transfer hostowany na własnym CDN Chase. Przekierowanie spełnia dwa cele: wzmacnia narrację marki JPMorgan z wyjściowego emaila oraz nie zostawia ofierze żadnego sygnału, że cokolwiek złośliwego się wydarzyło. Ofiara zamyka zakładkę, atakujący ma trwały dostęp do Microsoft 365.
Analiza kodu źródłowego - architektura backendu

Inspekcja DevTools strony phishingowej ujawnia dedykowany kit device code phishing z własnym API backendu - inaczej niż w THREAT-012, gdzie rolę backendu pełniło Cloudflare Workers:

// Etap 1: Inicjacja device code flow (backend atakującego → Microsoft)
async function sf() {
  const r = await fetch('/api/device/start', {
    method: 'POST',
    headers: { 'X-Antibot-Token': at }  // token anty-bot / wiązanie sesji
  });
  const d = await r.json();
  if (d.error) { document.getElementById('lds').classList.add('hidden'); }
  document.getElementById('dcs').classList.remove('hidden');
  pi = setInterval(ck, 3000);  // polling co 3 sekundy
}

// Etap 2: Polling do uzyskania tokenu
async function ck() {
  const d = await fetch(`/api/device/status/${sid}`).then(r => r.json());
  if (d.status === 'completed') {
    clearInterval(pi);
    // Token przejęty - przekierowanie na PDF JPMorgan jako kamuflaż
    window.location.replace('https://static.chasecdn.com/.../Wire_LA.pdf');
  }
}

Nagłówek X-Antibot-Token sugeruje logikę filtrowania botów - bezpośredni dostęp bez ważnego tokenu sesji prawdopodobnie zwraca stronę błędu, utrudniając automatyczne skanowanie przez badaczy i crawlery bezpieczeństwa.

Kod ujawnia też zachowanie dialogu autoryzacji Microsoft: gdy ofiara kończy przepływ, Microsoft wyświetla nazwę aplikacji jako "Microsoft Authentication Broker" i lokalizację urządzenia jako Stany Zjednoczone - kluczowy sygnał detekcji dla organizacji w Polsce, gdzie autoryzacja z US powinna być anomalią.

Porównanie z THREAT-012

Dwie kampanie Device Code Phishing w ciągu 6 dni. Wspólny mechanizm OAuth przy różnej infrastrukturze lure wymaga analizy porównawczej:

THREAT-012 (21 kwi 2026)

  • Lure: Microsoft Teams
  • Narracja: udostępniony dokument (Excel)
  • Dostarczenie: skompromitowane konto SendGrid
  • Backend: Cloudflare Workers (serverless)
  • Hop: skompromitowana domena .sg (leoch.sg)
  • Impersonacja aplikacji: Microsoft Office

THREAT-013 (27 kwi 2026)

  • Lure: JPMorgan Chase + Virtru
  • Narracja: płatność ACH (pilność finansowa)
  • Dostarczenie: nieznana infrastruktura
  • Backend: dedykowany serwer z /api/device/ API
  • Hop: tracker mailingowy (servidoraweb.net)
  • Impersonacja aplikacji: Microsoft Authentication Broker

Kluczowe różnice wskazują na oddzielnych aktorów lub kity: THREAT-012 nadużywa Cloudflare Workers (serverless, trudny do takedownu), THREAT-013 używa dedykowanego serwera z własnym API i logiką tokenu anty-bot. Jednocześnie 3-sekundowy interwał pollingu i użycie legitymnego dokumentu decoy po zakończeniu powtarzają się w obu kampaniach - potencjalnie wspólny bazowy kit z konfigurowalnymi szablonami lure.

Detekcja - reguły KQL

Microsoft Sentinel - autoryzacja device code z nieoczekiwanej lokalizacji:

// [HIGH] Device code authorization z nieoczekiwanej lokalizacji
// MITRE: T1528 · T1078.004
SigninLogs
| where TimeGenerated > ago(1d)
| where AuthenticationProtocol == "deviceCode"
| where ResultType == "0"  // Tylko udane logowania
| extend LocationCountry = tostring(LocationDetails.countryOrRegion)
| where LocationCountry != "Poland"  // Dostosuj do baseline organizacji
| summarize Count = count() by
    UserPrincipalName, LocationCountry, AppDisplayName,
    IPAddress, bin(TimeGenerated, 1h)
| order by TimeGenerated desc

Microsoft Sentinel - device code flow dla Microsoft Authentication Broker:

// [HIGH] Device code flow dla Microsoft Authentication Broker
// Prawdziwy device code prawie nigdy nie jest inicjowany dla tej aplikacji z przeglądarki
// MITRE: T1528
SigninLogs
| where TimeGenerated > ago(7d)
| where AuthenticationProtocol == "deviceCode"
| where AppDisplayName has_any ("Microsoft Authentication Broker", "Microsoft Office")
| where ResultType == "0"
| project TimeGenerated, UserPrincipalName, AppDisplayName,
    IPAddress, LocationDetails, DeviceDetail
| order by TimeGenerated desc

Microsoft Defender XDR - email z impersonacją JPMorgan / Virtru linkujący do zewnętrznej domeny:

// [MEDIUM] Impersonacja marki JPMorgan / Virtru w emailu z zewnętrznym linkiem
// MITRE: T1566.002
EmailEvents
| where TimeGenerated > ago(7d)
| where SenderFromAddress has_any ("jpmchase", "jpmorgan")
    and SenderFromDomain != "jpmchase.com"
| join kind=inner (
    EmailUrlInfo
    | where Url has_any ("servidoraweb", "deviceauth", "brentbr")
) on NetworkMessageId
| project TimeGenerated, Subject, SenderFromAddress,
    RecipientEmailAddress, Url, ThreatTypes

Uwaga operacyjna
SigninLogs z filtrem AuthenticationProtocol == "deviceCode" wymaga licencji Azure AD P2 lub Microsoft Sentinel z konektorem Entra ID. Dla organizacji bez P2, reguły consent grant z THREAT-012 pozostają najbardziej dostępną warstwą detekcji. Warto też alertować na AuditLogs dla nowych operacji Consent to application inicjowanych przez device code flow.

Mitigacja i reagowanie
1. Ograniczenie Device Code Flow przez Conditional Access
Entra ID → Security → Conditional Access → Nowa polityka → Docelowi użytkownicy: "Wszyscy" → Aplikacje klienckie: "Aplikacje mobilne i klienty stacjonarne" → Udziel: Blokuj
Uwaga: Przed włączeniem wyklucz konta serwisowe i znane urządzenia IoT.
Większość użytkowników korporacyjnych nigdy nie potrzebuje device code flow z poziomu przeglądarki - zablokowanie eliminuje ten wektor ataku. Wdrożyć najpierw w trybie Report-only i sprawdzić baseline false positive przed aktywacją.
2. Polityki zgody aplikacji - blokada zgody użytkownika
Entra ID → Enterprise Applications → Consent and permissions → Ustawienia zgody użytkownika: Nie zezwalaj na zgodę użytkownika
Wymusza, żeby wszystkie autoryzacje aplikacji OAuth przechodziły przez zatwierdzenie administratora. Eliminuje ryzyko wydania tokenu dla aplikacji zarejestrowanej przez atakującego bez wiedzy zespołu bezpieczeństwa.
3. Edukacja użytkowników - sygnały device code
Kody urządzenia powinny pojawiać się wyłącznie gdy użytkownik samodzielnie inicjuje logowanie na urządzeniu bez przeglądarki (Smart TV, narzędzie CLI). Jeśli kod pojawia się w emailu lub na stronie internetowej - to phishing. Kluczowy sygnał ostrzegawczy w dialogu autoryzacji Microsoft: lokalizacja urządzenia = Stany Zjednoczone przy logowaniu z Polski.
4. Jeśli token został wydany - reagowanie na incydent
Unieważnienie sesji: Entra ID → Użytkownicy → [Użytkownik] → Unieważnij wszystkie refresh tokeny
Audit zgód OAuth: Enterprise Applications → [Użytkownik] → Uprawnienia OAuth - przegląd ostatnich 90 dni
Sprawdź reguły skrzynki: Get-InboxRule -Mailbox [user] | Where {$_.ForwardTo -ne $null}
Sprawdź forwarding: Get-Mailbox [user] | Select ForwardingAddress, ForwardingSMTPAddress
Przegląd logów logowania: Szukaj dostępu z adresów IP z USA po zdarzeniu autoryzacji
