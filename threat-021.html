🔴
Bypass MFA — kradzież tokenu bez przechwycenia hasła
Atak nie przechwytuje hasła ofiary. Zamiast tego nadużywa mechanizmu OAuth 2.0 Device Authorization Grant (RFC 8628) w celu uzyskania ważnego tokenu dostępu Microsoft oraz 90-dniowego refresh tokenu. Token pozostaje aktywny nawet po zmianie hasła i daje pełen dostęp delegowany do konta Microsoft 365 ofiary.
🔍
Kluczowe ustalenie: Cloudflare Worker jako proxy OAuth polling
W odróżnieniu od wcześniejszych kampanii device code (THREAT-012, THREAT-013) przeglądarka ofiary nie komunikuje się bezpośrednio z endpointem tokenów Microsoft. Worker wykonuje całe odpytywanie po stronie serwera — atak jest niewidoczny w ruchu sieciowym przeglądarki.
Streszczenie

W dniu 2 czerwca 2026 roku do skrzynki firmowej dostarczono wiadomość phishingową podszywającą się pod zewnętrzny kontakt biznesowy. Email zawierał pojedynczy hiperłącze kierujące odbiorcę na darmową stronę hostowaną w serwisie Jimdo, imitującą portal weryfikacji dokumentów DocuSign. Kliknięcie przycisku „Review Document" przekierowywało do backendu na Cloudflare Workers, prezentującego wcześniej wygenerowany kod urządzenia OAuth i nakłaniającego ofiarę do uwierzytelnienia na prawdziwym portalu Microsoft.

Cloudflare Worker wykonywał całe odpytywanie endpointu tokenów Microsoft po stronie serwera — żaden ruch OAuth nie był widoczny w przeglądarce ofiary. Po pomyślnym wydaniu tokenu Worker przekierowywał ofiarę na microsoft.com jako przykrywkę. Przeglądarka oznaczyła domenę phishingową jako niebezpieczną (czerwony ekran ostrzeżenia) przed załadowaniem strony — atak został powstrzymany.

Łańcuch Ataku
01
Dostarczenie emaila — zewnętrzny nadawca, zanonimizowany
Wiadomość phishingowa została wysłana z zewnętrznego adresu z tematem: „Follow up on confidential updated documented projected plans for your concern feedback". Treść wiadomości była minimalna — pojedyncza linia tekstu z hiperłączem „See Document here" oraz ogólny blok stopki z danymi kontaktowymi. Brak załącznika.

Niskiej jakości temat wiadomości i ogólnikowa treść wskazują na kampanię masową (spray-and-pray) a nie ukierunkowany spearphishing. Domena nadawcy została w pełni zanonimizowana zgodnie z polityką writeupów.
02
Strona pośrednicząca — darmowy hosting Jimdo (pillabetelproject-3.jimdosite.com)
Hiperłącze prowadziło do strony na pillabetelproject-3[.]jimdosite[.]com — darmowej witryny utworzonej na platformie Jimdo. Strona wyświetlała sklonowany interfejs DocuSign pokazujący dokument „Non-Disclosure Agreement" (NDA-2026-Final.pdf · 4 strony) z widocznym przyciskiem „Review Document" i stopką „© 2026 DocuSign, Inc."

Jimdo jako warstwa pośrednicząca: darmowe kreatory stron są coraz częściej nadużywane jako przeskoki przekierowań, ponieważ działają na legalnych domenach z ważnymi certyfikatami TLS, co zmniejsza prawdopodobieństwo ich oznaczenia przez narzędzia skanujące linki. Sama strona nie zawierała złośliwego kodu — jedynie przekierowywała na backend Worker. Metadane og:description strony Jimdo zawierały tekst „NEW DOCUMENT RECEIVED".
03
Cloudflare Worker — klon DocuSign + dostarczenie kodu urządzenia
Kliknięcie „Review Document" otwierało hxxps://int-ems[.]c-66idsk0d[.]workers[.]dev. Strona kontynuowała podszywanie się pod DocuSign — w kroku „Verify Identity" wyświetlała wcześniej wygenerowany kod OAuth urządzenia (A4S5P2X46) z przyciskiem „Copy code" oraz CTA „Sign in with Microsoft".

Kod urządzenia był statycznie osadzony w HTML strony — atakujący wygenerował go przed wizytą ofiary, wywołując endpoint /oauth2/v2.0/devicecode Microsoft. Kliknięcie „Sign in with Microsoft" otwierało w oknie popup prawdziwy portal aka.ms/devicelogin, gdzie ofiara byłaby proszona o wpisanie kodu i uwierzytelnienie kontem Microsoft.
04
Proxy Polling — Worker odpytuje Microsoft w imieniu przeglądarki
Równocześnie JavaScript osadzony na stronie Workera rozpoczynał odpytywanie własnego originu poprzez POST co 4 sekundy, przesyłając surową wartość device_code jako parametr formularza (dc=EBgABIQE...). Backend Worker przekazywał te żądania do endpointu tokenów Microsoft po stronie serwera.

To kluczowa różnica względem THREAT-012/013: w poprzednich kampaniach przeglądarka ofiary odpytywała Microsoft bezpośrednio. Tutaj ruch OAuth Microsoft jest niewidoczny w przeglądarce ofiary — odbywa się wyłącznie w infrastrukturze Cloudflare. Polling działał przez maksymalnie 15 minut (timeout 900 sekund) z interwałem 4 sekund. Po odebraniu tokenu Worker zwracał {"r":true} i strona przekierowywała na microsoft.com.

Analiza JavaScriptu — skrypt frontendu Workera

Poniższy skrypt został wyodrębniony ze strony Cloudflare Worker. Identyfikatory elementów są zaciemnione losowymi ciągami znaków (muqilok, letojilu, wemef itp.), co utrudnia analizę bez dostępu do pełnego kodu HTML. Kluczowe zachowania opisano w komentarzach inline:

// Skrót document.getElementById + zmienna stanu dla interwału
var _fb6463 = function(i) { return document.getElementById(i) }, _712ba;

// Przycisk kopiowania kodu — kopiuje kod urządzenia do schowka, pokazuje "✓ Copied" przez 1,8s
_fb6463("gifimid").onclick = function() {
  navigator.clipboard.writeText(_fb6463("wurawa").textContent.trim())
    .then(function() {
      _fb6463("gifimid").textContent = "✓ Copied";
      setTimeout(function() {
        _fb6463("gifimid").textContent = "Copy code";
      }, 1800);
    })
};

// Przycisk logowania — otwiera aka.ms/devicelogin w popup, następnie uruchamia proxy polling
_fb6463("welup").onclick = function() {
  _fb6463("welup").disabled = true;
  _fb6463("dumoziz").textContent = "Waiting for sign-in…";

  // Otwiera prawdziwy portal logowania Microsoft w oknie popup
  window.open("https://aka.ms/devicelogin", "_blank", "popup,width=700,height=650");

  var t = Date.now();
  _712ba = setInterval(function() {

    // Timeout 15 minut (9e5 ms = 900s — zgodnie z TTL kodu urządzenia Microsoft)
    if (Date.now() - t > 9e5) { clearInterval(_712ba); return; }

    // POST do własnego originu Workera — dc= zawiera surowy device_code z Microsoft
    // Worker przekazuje żądanie do endpointu tokenów Microsoft po stronie serwera
    fetch("", {
      method: "POST",
      headers: { "Content-Type": "application/x-www-form-urlencoded" },
      body: "dc=EBgABIQEAAAAdDD7n..."  // skrócone — pełny device_code (~600 znaków)
    })
    .then(function(r) { return r.json() })
    .then(function(j) {
      // Worker zwraca {"r":true} gdy Microsoft wyda token
      if (j.r) {
        clearInterval(_712ba);
        // Przykrywka — przekierowanie ofiary na microsoft.com po przechwyceniu tokenu
        location.href = "https://www.microsoft.com/en-us/";
      }
    })
    .catch(function() {});

  }, 4e3);  // Odpytywanie co 4 sekundy
};
