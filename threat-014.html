Custom Proof-of-Work Bot Protection
Phishing kit implementuje własny mechanizm PoW oparty na SHA-256 - nie jest to prawdziwy Cloudflare, jedynie wizualna imitacja. Serwer wydaje nonce z poziomem trudności, przeglądarka ofiary przeprowadza wielorundowe hashowanie SHA-256 na wszystkich rdzeniach CPU, rozwiązanie jest weryfikowane przez backend. Cel: eliminacja botów, crawlerów i automatycznych skanerów URL.
Executive Summary

W dniu 29 kwietnia 2026 roku użytkownik firmowy zgłosił podejrzany email podszywający się pod serwis Intuit QuickBooks. Email zawierał groźbę debitu z konta bankowego w ciągu 1–2 dni roboczych i nakłaniał do kliknięcia linku w celu zakwestionowania transakcji. Rzeczywisty URL prowadził przez serwis śledzący Postmark do domeny wrenixo[.]cfd - dedykowanej infrastruktury phishingowej obsługującej cały atak.

Analiza wykazała zaawansowany, wieloetapowy phishing kit z własną implementacją Proof-of-Work jako filtrem botów, perfekcyjną kloną interfejsu Intuit oraz łańcuchem ekstrakcji danych progresywnie eskalującym zakres zbieranych informacji: credentials → banking data → SSN → skan dokumentu tożsamości. Każda ofiara otrzymuje unikalne URL generowane per sesja, które wygasają po przetworzeniu - co utrudnia retroaktywną analizę i blokowanie po sygnaturze URL.

Po zebraniu wszystkich danych ofiara jest przekierowywana na prawdziwą stronę accounts.intuit.com z komunikatem o błędzie, sugerując pomyłkę podczas logowania. Mechanizm ten maskuje fakt kradzieży i opóźnia reakcję ofiary.

Łańcuch ataku - 6 etapów
01
Email Delivery - skompromitowane konto akademickie (spup.edu.ph)
Email został nadany z adresu należącego do Saint Paul University Philippines - filipińskiej uczelni wyższej. Konto nadawcy jest najprawdopodobniej przejęte lub nadużywane przez aktora kampanii. Domena akademicka o ustalonej reputacji obniża prawdopodobieństwo detekcji przez filtry antyspamowe oparte na reputacji domeny.

Wyświetlany nadawca: QuickBooks <[redacted][@]spup.edu.ph>
Wyświetlany link: QuickBooks.com/login/account/payout/case#55142661711/php (anchor text - nie rzeczywisty URL)
Rzeczywisty URL po hover: hxxps://track.pstmrk[.]it/3s/track.pstmrk[.]it/... → redirect chain przez wrenixo[.]cfd

Treść email operuje na mechanizmach urgency i financial threat: informuje o otwartym sporze dotyczącym niedostarczonej płatności oraz grozi obciążeniem konta bankowego odbiorcy w ciągu 1–2 dni roboczych bez natychmiastowej reakcji. Case number (#55141661711) nadaje emailowi pozory oficjalności.
02
Tracking Redirect - Postmark (pstmrk.it) → wrenixo[.]cfd
Link w emailu przechodzi przez legalny serwis śledzenia emaili Postmark (track.pstmrk[.]it). Postmark jest popularną platformą email marketingową - jej domeny cieszą się wysoką reputacją i rzadko są blokowane przez filtry URL. Atakujący nadużywa jej infrastruktury wyłącznie jako pierwszego hoppa redirect chain, ukrywając docelową domenę phishingową do momentu kliknięcia.

URL zawiera zakodowane parametry: session, cid (campaign/client identifier) oraz token jednorazowy przypisany do konkretnej ofiary. Parametry te umożliwiają backendowi śledzenie każdego kroku sesji oraz serwowanie unikalnych URL per etap.

Końcowy redirect trafia do wrenixo[.]cfd - domeny zarejestrowanej w TLD .cfd (Call For Dream), popularnym wśród phisherów ze względu na niską cenę rejestracji.
03
Bot Filter - Custom SHA-256 Proof-of-Work (imitacja Cloudflare Turnstile)
Pierwsza strona prezentuje interfejs wizualnie identyczny z Cloudflare Turnstile challenge - logo tarczy, przycisk "I am human" / "Verify Now", Ray ID. W rzeczywistości jest to całkowicie własna implementacja, nie mająca związku z Cloudflare.

Po kliknięciu przycisku przeglądarka wykonuje następujący przepływ:

// 1. Pobranie challenge z serwera GET /c4a8b2 → { nonce, difficulty, rounds, cid } // 2. SHA-256 PoW na wszystkich rdzeniach CPU (Web Workers) while(true) { hash = SHA256(nonce + i); for(j=0; j<rounds; j++) hash = SHA256(hash); // domyślnie 10 000 rund if((hash[0..3] as uint32) < 2^32 / 2^difficulty) { postMessage({ solution: i }); return; } i += numWorkers; } // 3. Weryfikacja rozwiązania i pobranie redirect URL POST /v9f3e1 body: cid=...&pow=... → { success: true, redirect: "..." } // 4. Sztuczne opóźnienie min. 2000ms dla wiarygodności if (elapsed < 2000) await sleep(2000 - elapsed);
Cel mechanizmu: odfiltrowanie automatycznych skanerów URL, headless browsers bez JS, crawlerów antywirusowych. Tylko prawdziwy użytkownik z działającą przeglądarką przechodzi przez ten etap i uzyskuje dostęp do kolejnych warstw phishingu.
04
Credential Harvest - sklonowany interfejs Intuit QuickBooks
Po przejściu bot filtru ofiara trafia na stronę hostowaną na _wildcard_[.]wrenixo[.]cfd - perfekcyjną kopię interfejsu logowania Intuit/QuickBooks. Strony zawierają prawdziwy version string (data-shell-version="6.227.1-master-bld.419-7d7beb1d-419"), kompletne arkusze CSS z animations Intuit Design System, oraz atrybuty data-testid identyczne z produkcyjnym kodem Intuit - co sugeruje klonowanie przez dedykowane narzędzie (HTTrack lub podobne).

Formularz logowania składa się z dwóch kroków:
Krok 1: Email / User ID → akceptuje dowolną wartość
Krok 2: Hasło → akceptuje dowolną wartość → redirect na etap 5

Brak weryfikacji wprowadzonych danych potwierdza, że priorytetem jest zebranie jak największej liczby pól PII, nie uwierzytelnianie wobec prawdziwego serwera Intuit.
05
Extended Data Harvest - dane bankowe, SSN/EIN, skan dokumentu ID
Po "zalogowaniu" ofiara jest prowadzona przez dwa dodatkowe formularze, utrzymując pełne branding Intuit:

Formularz "Confirm Your Information":
 → Account Number (numer konta bankowego)
 → Routing Number (numer rozliczeniowy ABA)
 → SSN / EIN (numer ubezpieczenia społecznego lub identyfikator pracodawcy)
 → Date of Birth
 → Business Name

Formularz "ID Upload":
 → Zdjęcie przodu dokumentu tożsamości (prawo jazdy, state ID)
 → Zdjęcie tyłu dokumentu tożsamości
 → Akceptuje PDF, JPG, PNG

Test z fałszywym student ID (placeholder bez prawdziwych danych) wykazał, że system akceptuje dowolne pliki bez weryfikacji - upload jest przekazywany bezpośrednio do backendu atakującego.
06
Exit Redirect - prawdziwy accounts.intuit.com + komunikat o błędzie
Po zebraniu wszystkich danych ofiara jest przekierowywana na prawdziwą stronę accounts.intuit.com. Strona wyświetla komunikat "Double check your info - We can't find an account with what you entered" - sugerując, że ofiara popełniła błąd podczas logowania na oryginalnej stronie.

Mechanizm ten pełni dwie funkcje: maskuje fakt kradzieży (ofiara sądzi, że nigdy nie trafiła na phishingową stronę) oraz opóźnia reakcję - ofiara może podjąć kolejną próbę logowania zamiast zgłosić incydent. Jest to klasyczna technika lure completion stosowana w zaawansowanych kitach phishingowych.
Analiza techniczna phishing kitu

Infrastruktura sesji i URL per ofiara

Każda ofiara otrzymuje unikalny zestaw URL generowanych przez backend z parametrami session i cid. Kolejne etapy ataku (login, hasło, dane bankowe, ID upload) serwowane są pod oddzielnymi, losowymi ścieżkami w formacie /qbo/{hash_a}/{hash_b}. URL wygasa po przetworzeniu sesji - próba dostępu do zużytego URL zwraca 404 Not Found z serwerem Apache. Mechanizm utrudnia retroaktywną analizę, blokowanie po URL przez systemy bezpieczeństwa oraz automatyczne skanowanie przez honeypoty.

Steganografia HTML - ukryte session tokeny w stopce

W kodzie źródłowym każdej strony kitu, w tekście stopki z informacją prawną Intuit, osadzone są ukryte tokeny przy użyciu niewidocznych fontów:

Intuit, Quic<font style='color:transparent;font-size:0px'>O41711</font>kBooks, QB, Turb<font style='color:transparent;font-size:0px'>E89526</font>oTax... → Wizualnie: "Intuit, QuickBooks, QB, TurboTax..." → W DOM: ukryte kody per strona: O41711, E89526, Z30122, 664024, T95071

Każda strona kitu zawiera unikalny zestaw takich tokenów (5 fragmentów per strona, różne kody dla etapu login/hasło/dane bankowe). Prawdopodobne zastosowania: weryfikacja przez backend, która wersja strony dotarła do ofiary; antyanalizowe znaczniki utrudniające kopiowanie/modyfikację kitu; lub śledzenie konwersji per etap per kampania.

Wildcard subdomain jako mechanizm generowania URL

Hosting odbywa się na _wildcard_[.]wrenixo[.]cfd - nazwa subdomeny wskazuje na certyfikat wildcard (*[.]wrenixo[.]cfd), co pozwala atakującemu na serwowanie treści pod dowolną subdomeną bez osobnej rejestracji certyfikatu TLS. Każda ofiara może w teorii otrzymać unikalną subdomenę, co drastycznie utrudnia blokowanie po nazwie hosta.

Ocena sophistication kitu: Kombinacja własnego PoW bot filtru, sesyjnych URL jednorazowych, steganografii HTML i wildcard subdomains wskazuje na custom-developed kit lub mocno zmodyfikowany zaawansowany framework (klasy Evilginx/Modlishka). Nie jest to gotowy kit z darkweb - wymagał dedykowanego nakładu pracy developmentu.

Możliwości detekcji

Ze względu na brak złośliwego ładunku po stronie klienta, detekcja opiera się wyłącznie na analizie email, reputacji URL i zachowania użytkownika.

Email Gateway:
Blokowanie wiadomości zawierających Postmark redirect URL (track.pstmrk[.]it) prowadzący do domen [.]cfd. Analiza rozbieżności między wyświetlanym anchor textem linku a rzeczywistym href (anchor: QuickBooks.com/..., href: external tracking domain). Reguła na mismatched sender domain: From: QuickBooks + nadawca spoza intuit.com.
Proxy / URL Filtering:
Blokowanie TLD .cfd jako kategorii wysokiego ryzyka. Wykrywanie PoW challenge pattern: żądania GET /[a-z0-9]{6} zwracające JSON z polami nonce, difficulty, rounds, cid - unikalny fingerprint backendu kitu.
Browser / Endpoint:
Wykrywanie tworzenia wielu Web Workers uruchamiających intensywne operacje SHA-256 (charakterystyczny profil CPU na stronie phishingowej PoW). SmartScreen / Safe Browsing flagowanie domeny wrenixo[.]cfd.
Ograniczenia detekcji:
Postmark jako legalny serwis ma wysoką reputację i rzadko jest blokowany. Wildcard subdomeny utrudniają blokowanie po hostname. URL jednorazowe (wygasające po użyciu) uniemożliwiają skuteczne blokowanie po pełnym URL w trybie retroaktywnym. Strony kitu przechodzą podstawową weryfikację TLS (zielona kłódka). Kod HTML jest niemal identyczny z produkcyjnym Intuit - detekcja po content fingerprint jest trudna.
Mitigation & Rekomendacje

Natychmiastowe: Zablokowanie domeny wrenixo[.]cfd i wszystkich subdomen na poziomie DNS i proxy. Zgłoszenie domeny do Abuse Contact (Cloudflare, rejestrator domeny .cfd) oraz do CERT Polska (incydent.cert.pl). Poinformowanie użytkownika o incydencie i weryfikacja, czy wprowadził realne dane (ze względu na anonimizację sesji potwierdzono użycie fałszywych danych testowych).

Polityki email: Wdrożenie lub zaostrzenie reguły blokującej mismatched display name (QuickBooks/Intuit) vs domena nadawcy. Reguła na linki prowadzące przez Postmark do TLD o wysokim ryzyku (.cfd, .xyz, .top, .pw). Szkolenie użytkowników z weryfikacji rzeczywistego URL linku (hover przed kliknięciem).

Świadomość użytkowników: Legalne serwisy finansowe (Intuit, banki) nigdy nie proszą o upload dokumentu tożsamości przez email. Żądanie SSN/EIN, routing number i skanu ID w jednej sesji jest jednoznacznym wskaźnikiem fraudu - niezależnie od wiarygodności wizualnej strony. Adres URL w pasku przeglądarki jest zawsze miarodajnym wskaźnikiem: intuit.com vs wrenixo.cfd.

Threat Intelligence: Domena wrenixo[.]cfd powinna zostać dodana do wewnętrznej listy blokowania IoC i udostępniona do platform threat intelligence. Monitorowanie pojawienia się nowych domen z tym samym wzorcem URL (/qbo/{hex}/{hex}) i podobnym fingerprintem PoW backendu (/c4a8b2, /v9f3e1).
