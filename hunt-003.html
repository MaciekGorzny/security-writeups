Threat-Informed Detection - nie incydent produkcyjny
Niniejszy wpis dokumentuje proaktywne budowanie detekcji na podstawie advisory Microsoft MSRC oraz publicznego PoC opublikowanego przez badacza Nightmare Eclipse (16 czerwca 2026) dotyczącego luki elevation-of-privilege w Microsoft Defender, znanej jako RoguePlanet. Reguły KQL zostały opracowane i wdrożone proaktywnie, przed wystąpieniem jakiegokolwiek potwierdzonego incydentu w środowisku.
ℹ️
Metodologia
Analiza advisory i dostępnych źródeł TI → identyfikacja artefaktów behawioralnych możliwych do zaobserwowania mimo braku deterministycznego PoC → priorytetyzacja sygnałów odpornych na race-condition (czyli nie zależnych od tego, czy konkretna próba eksploatacji się powiodła) → budowa reguł KQL → identyfikacja ślepych punktów → weryfikacja w lab.
⚠️
Kluczowa obserwacja: brak łatki silnika w momencie publikacji
Microsoft potwierdził podatność, ale nie podał wersji silnika Malware Protection Engine, w której zostanie naprawiona. Posiadanie najnowszego pakietu intelligence Defendera nie oznacza automatycznie remediacji — aktualizacje silnika/platformy mają inny cykl wydawniczy niż codzienne sygnatury. Do czasu wydania łatki priorytetem pozostaje detekcja behawioralna i ograniczenie blast radius.
🧪
Notatka z lab
Publiczny PoC testowany w naszym środowisku laboratoryjnym ok. dwóch tygodni przed publikacją tego wpisu — już wtedy obserwowaliśmy problemy z konsystentnym działaniem exploita. Jest to zgodne z naturą luki: autor PoC sam opisuje ją jako race condition o zmiennej skuteczności w zależności od konkretnej maszyny i stanu systemu. Niezawodność exploita w realnych środowiskach produkcyjnych — różnorodnych pod względem sterowników, oprogramowania trzeciego i obciążenia systemu — pozostaje source'em niepewności, dlatego detekcja celuje w sygnały odporne na to, czy dana próba zakończyła się sukcesem czy nie.
Kontekst zagrożenia

CVE-2026-50656 to luka elevation-of-privilege w Microsoft Malware Protection Engine — komponencie silnika Microsoft Defender, publicznie nazwana RoguePlanet. Opublikowana przez Microsoft 16 czerwca 2026, sklasyfikowana jako CWE-59 — Improper Link Resolution Before File Access (Link Following).

CVSS 3.1 base score 7.8 High (wektor lokalny, niska złożoność, wymagane niskie uprawnienia, brak interakcji użytkownika). NVD ocenia attack complexity wyżej (7.0, high complexity) — rozbieżność wynika z probabilistycznej natury luki, opisanej szerzej poniżej. CISA-ADP klasyfikuje exploitation jako "public proof-of-concept", bez potwierdzonej aktywnej eksploatacji w momencie publikacji advisory.

Autor i kontekst: badacz działający pod pseudonimem Nightmare Eclipse (alias Chaotic Eclipse) publikuje od marca 2026 serię zero-day'ów dla produktów Microsoft, najwyraźniej w odpowiedzi na spór dotyczący procesu disclosure z firmą. RoguePlanet jest czwartą luką tego autora po BlueHammer (CVE-2026-33825), RedSun (CVE-2026-41091) i UnDefend (CVE-2026-45498) — część z nich była eksploatowana in-the-wild zanim doczekała się łatki. Ten wzorzec ma znaczenie operacyjne: traktujemy komponenty bezpieczeństwa (sam silnik Defendera) jako potencjalny cel ataku, a nie wyłącznie jako linię obrony.

Mechanizm ataku
PREREQ Wymagany foothold + montowanie obrazu ISO/VHD T1204 (kontekst)
Luka wymaga, by atakujący posiadał już dostęp z niskimi uprawnieniami na hoście (phishing, dropper, skradzione poświadczenia). Publiczny PoC wymaga zamontowania obrazu ISO/VHD jako standardowy użytkownik jako etap przygotowawczy. Autor PoC zaznacza, że na Windows Server ten krok nie działa, ponieważ standardowi użytkownicy nie mogą montować obrazów — wskazuje to jednak na ograniczenie samego PoC, nie na odporność platformy: wg autora wszystkie wersje Windows Server pozostają teoretycznie podatne, exploit wymagałby tylko przeprojektowania tego etapu.
RACE TOCTOU race condition w logice link-resolution silnika T1068 · CWE-59
Sedno podatności: silnik Malware Protection Engine wykonuje uprzywilejowaną operację plikową bez odpornego na wyścig sprawdzenia, czy ścieżka nie została przekierowana (np. przez reparse point / junction) pomiędzy momentem weryfikacji a momentem użycia (time-of-check to time-of-use). Eksploatacja jest probabilistyczna — zależna od trafienia w wąskie okno czasowe, co tłumaczy zmienną skuteczność obserwowaną zarówno przez autora PoC, jak i w naszych własnych testach lab. Pierwotnie luka umożliwiała zdalny wektor przez spreparowane pliki .vhd(x) na udziałach SMB; działania hardeningowe Microsoftu z maja 2026 zamknęły ten wektor, ograniczając obecny publiczny exploit do lokalnej eskalacji uprawnień.
IMPACT Spawn powłoki SYSTEM T1059 · T1211
Przy udanym trafieniu w okno wyścigu, silnik Defendera (uruchomiony z wysokimi uprawnieniami) staje się rodzicem nowego procesu powłoki działającego w kontekście SYSTEM. Istotne dla modelowania detekcji: exploit działa niezależnie od stanu real-time protection Defendera — wyłączenie lub włączenie agenta nie jest skutecznym obejściem ani środkiem zaradczym.
Reguły Detekcyjne KQL
RULE-01 [Privilege Escalation / Defense Evasion - T1068 · T1211] MsMpEng.exe jako rodzic powłoki DeviceProcessEvents
Najsilniejszy sygnał w całym zestawie. MsMpEng.exe nigdy w normalnych warunkach operacyjnych nie powinien spawnować interaktywnej powłoki — sam fakt wystąpienia takiej relacji rodzic-dziecko jest wysoce anomalny, niezależnie od konta, na którym uruchomiony jest proces potomny. Celowo pominięto filtrowanie po AccountName — sztywny warunek na SYSTEM zawężałby pole widzenia bez realnego zysku i mógłby przeoczyć przypadki, gdzie race condition produkuje proces potomny w trakcie tranzycji uprawnień, zanim token w pełni "dotrze" do SYSTEM.
Wysoka pewność T1068 · T1211
// RULE-01: RoguePlanet (CVE-2026-50656) — MsMpEng spawning a shell // Rationale: MsMpEng.exe never legitimately spawns interactive shells, // regardless of the resulting process's account context. AccountName is // intentionally NOT used as a filter — only retained in output for triage — // since the parent-process relationship alone is the strong signal here. DeviceProcessEvents | where InitiatingProcessFileName =~ "MsMpEng.exe" | where FileName has_any ("cmd.exe", "powershell.exe", "pwsh.exe", "conhost.exe", "cscript.exe", "wscript.exe") | project Timestamp, DeviceName, InitiatingProcessFileName, InitiatingProcessId, FileName, ProcessCommandLine, AccountName, InitiatingProcessAccountName // FP considerations: practically none in baseline telemetry — high-confidence // rule regardless of AccountName value. Can be deployed fleet-wide without staged rollout.
RULE-02 [Preparation - T1204] Pętla montowania obrazów ISO/VHD (CLI) DeviceProcessEvents
Publiczny PoC wymaga zamontowania obrazu jako użytkownik niskouprzywilejowany jako etap przygotowawczy poprzedzający wyzwolenie wyścigu. Reguła pokrywa trzy ścieżki CLI: PowerShell (Mount-DiskImage), mountvol oraz diskpart (select vdisk file / attach vdisk) — diskpart jest oczywistą alternatywą, którą każdy choć trochę doświadczony atakujący weźmie pod uwagę przy automatyzacji pętli prób. Sygnał sam w sobie jest słaby i kontekstowy (legalne montowanie ISO jest powszechne), dlatego threshold wymaga wielokrotnych prób w krótkim czasie — przydatny głównie jako składowa korelacji w RULE-03.
Średnia pewność (kontekstowa) T1204
// RULE-02: RoguePlanet precursor — local image mount loop (CLI paths) // Rationale: covers PowerShell, mountvol, and diskpart as primary CLI mounting // paths used to stage the exploit prerequisite. Explorer-driven double-click // mounts are NOT captured here — see RULE-02B. DeviceProcessEvents | where FileName in~ ("powershell.exe", "pwsh.exe", "cmd.exe", "diskpart.exe") | where ProcessCommandLine has_any ( "Mount-DiskImage", "mountvol", "-ImagePath", "select vdisk file", "attach vdisk" ) | summarize MountAttempts = count(), FirstSeen = min(Timestamp), LastSeen = max(Timestamp) by DeviceName, AccountName = InitiatingProcessAccountName | where MountAttempts >= 5 // FP considerations: software deployment teams, ISO-based imaging workflows, // sysadmin use of diskpart for unrelated disk management — expect baseline noise, // tune threshold per environment, recommend allowlisting known deployment accounts.
RULE-02B [Preparation - T1204] Montowanie obrazu przez Eksplorera (dwuklik) DeviceFileEvents
Ślepy punkt RULE-02: dwuklik na pliku .iso w Eksploratorze montuje obraz przez wewnętrzny mechanizm shell32/virtdisk, bez generowania osobnej linii poleceń — żaden z warunków RULE-02 tego nie złapie. Reguła pokrywa ten wariant dwoma podejściami: (A) korelacja po explorer.exe jako proces inicjujący i artefaktach plikowych — dostępna od razu, ale sygnał słaby i szumowy; (B) docelowo, dziennik operacyjny Microsoft-Windows-VHDMP-Operational dający bezpośredni sygnał faktycznego zdarzenia attach/mount — wymaga dociągnięcia przez AMA, co nie jest domyślnie zbierane w naszym środowisku. Dodano celowo, by pokryć przypadek innego PoC korzystającego z tej samej luki przez inny wektor uruchomienia.
Niska pewność (Option A) / Średnia (Option B) T1204
// RULE-02B: Explorer-driven (double-click) ISO/VHD mount // Rationale: explorer.exe mounts via shell32/virtdisk internally — no CLI command // line is generated, so RULE-02 cannot see this path. // --- Option A: DeviceFileEvents correlation (available now, weak signal) --- DeviceFileEvents | where InitiatingProcessFileName =~ "explorer.exe" | where FolderPath matches regex @"^[A-Z]:\\\\$" or FileName endswith ".iso" or FileName endswith ".vhd" or FileName endswith ".vhdx" | project Timestamp, DeviceName, InitiatingProcessAccountName, InitiatingProcessFileName, FileName, FolderPath, ActionType // FP considerations: explorer.exe touches .iso files for many legitimate reasons // (browsing downloads, copying images). Use as secondary correlation, not standalone alert. // --- Option B (preferred): VHDMP-Operational event log, ingested as custom table --- // Requires forwarding Microsoft-Windows-VHDMP-Operational via AMA custom XML collection. // VHDMP_CL // | where EventID_d in (1, 12) // verify exact IDs against your build before deploying // | where InitiatingProcessAccountName !has "SYSTEM" // | project TimeGenerated, Computer, EventID_d, AccountName_s, FileName_s // FP considerations: legitimate installer/driver-package ISO mounting will also // appear here — baseline normal mount volume per host before alerting.
RULE-03 [Privilege Escalation - T1068] Korelacja: montowanie obrazu + eskalacja konta DeviceProcessEvents (join)
Łączy słaby sygnał z RULE-02 (pętla montowania) z silniejszym sygnałem zmiany konta: powłoka uruchomiona w oknie 5 minut od ostatniego montowania, działająca na innym (wyższym) koncie niż to, które wykonało montowanie. Kluczowa decyzja projektowa: zamiast sztywno filtrować po AccountName == "SYSTEM" (co przeoczyłoby inne formy eskalacji i zakładałoby z góry który dokładnie warunek szukamy), reguła wykrywa samą zmianę konta między etapem przygotowawczym a spawnem powłoki — SpawnAccount != AccountName. Z procesu inicjującego usunięto diskpart.exe, by uniknąć self-matchingu: bez tego diskpart użyty do zamontowania obrazu pasowałby do obu nóg joina i generował alert przy każdym pojedynczym użyciu, niezależnie od faktycznej eskalacji.
Wysoka pewność (po korekcie) T1068
// RULE-03: RoguePlanet correlation — mount loop followed by account-escalated shell // Rationale: combines weak signal (mount loop) with strong signal (shell spawned // under a DIFFERENT account than the one that performed the mount) within a 5-minute // window. diskpart.exe intentionally excluded from SuspiciousSpawns to avoid // self-matching against its own mount-triggering invocation in MountLoops. let MountLoops = DeviceProcessEvents | where ProcessCommandLine has_any ("Mount-DiskImage", "mountvol", "-ImagePath", "select vdisk file", "attach vdisk") | summarize MountCount = count(), LastMount = max(Timestamp) by DeviceName, AccountName = InitiatingProcessAccountName; let SuspiciousSpawns = DeviceProcessEvents | where FileName has_any ("powershell.exe", "pwsh.exe", "cmd.exe", "conhost.exe") | where InitiatingProcessFileName !in~ ("services.exe", "svchost.exe", "wininit.exe", "taskeng.exe", "schtasks.exe") | project DeviceName, DeviceId, SpawnTime = Timestamp, InitiatingProcessFileName, FileName, ProcessCommandLine, SpawnAccount = AccountName; MountLoops | where MountCount >= 3 | join kind=inner SuspiciousSpawns on DeviceName | where SpawnTime between (LastMount .. LastMount + 5m) | where SpawnAccount != AccountName | project DeviceName, DeviceId, MountingAccount = AccountName, SpawnAccount, MountCount, LastMount, SpawnTime, InitiatingProcessFileName, FileName, ProcessCommandLine // FP considerations: legitimate account-switch scenarios (RunAs, scheduled tasks // under service accounts, UAC elevation) will still appear here. SpawnAccount != // AccountName is the core anti-noise filter — same-user mount-then-shell, the // overwhelming majority of legitimate activity, is excluded entirely.
